# 方案：首个可发布版本整理

## 背景

当前仓库已经具备：

- 可用的 Composer 包结构
- Pull / Push 同步能力
- 本地宿主验证记录
- 完整相关测试迁移与通过记录
- 发布前检查清单

下一步要解决的不是功能缺口，而是“如何形成第一个可发布版本”的口径与步骤。

## 目标

- 明确首个版本号策略
- 明确 tag 约定
- 明确 Packagist 发布前需要检查的内容
- 把这些信息沉淀成稳定文档，便于后续重复执行

## 推荐方案

### 1. 采用 pre-1.0 起步

建议首个版本使用：

- `v0.1.0`

原因：

- 当前包仍与 BookStack 宿主内部运行时强耦合
- 已验证可用，但兼容边界还未完全收敛
- 用 `0.x` 比直接宣称 `1.0.0` 更符合当前风险等级

### 2. tag 约定

建议使用带 `v` 前缀的语义化版本：

- `v0.1.0`
- `v0.1.1`
- `v0.2.0`

### 3. Packagist 发布前要求

至少确认：

- 仓库公开可访问或 Packagist 可访问
- 默认分支与 tag 策略清晰
- `composer.json` 中包名、autoload、license 正确
- README 中安装方式与兼容性说明不夸大
- 发布说明清楚写出当前依赖 BookStack 宿主

### 4. 发布文档化

建议补一份单独文档，包含：

- 推荐首发版本号
- 版本升级规则
- tag 示例命令
- Packagist 发布前检查项
- 发布后验证建议

## 不纳入本轮范围

- 自动发版流水线
- 多版本 BookStack 兼容矩阵
- 强行承诺稳定公共 API

## 预期结果

- 形成第一版版本策略文档
- 发布前可以按文档逐条执行
- 后续每次发布都能复用同一套步骤

## 结论

- 状态：执行中
- 当前推荐：先沉淀首发版本策略与发布步骤文档
