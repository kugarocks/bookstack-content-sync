# 方案：发布前整理与基础自动化

## 背景

当前扩展包已经完成：

- Pull / Push 核心代码迁移
- 宿主 BookStack 本地安装验证
- 最小 pull 实跑验证
- push plan 与 no-change push execute 验证
- 所有直接相关的单元测试与集成测试迁移

目前主要缺口已经从“功能可用性”转向“后续维护与发布准备”。

## 目标

- 同步并修正当前 `todo/02-progress.md` 的真实状态
- 明确下一阶段的收尾事项，避免计划文档继续滞后
- 补最基础但高收益的发布辅助材料，并保留本地验证优先的节奏

## 建议范围

### Phase 10.1：同步项目状态

- 更新 `todo/02-progress.md`
- 把当前阶段从“规划中”调整为更符合现状的状态
- 把已经完成的测试迁移、命令验证、文档补充写入进度
- 重写“下一步建议”，对齐当前真实情况

### Phase 10.2：补发布辅助文档

可以补充一份发布前检查清单，例如：

- 版本号与 tag 策略
- Packagist 发布前检查项
- 与宿主 BookStack 的兼容性说明
- 本地与宿主验证步骤

## 不纳入本轮范围

本轮先不做：

- 抽离与 BookStack 的内部耦合
- 跨多个 BookStack 版本的大规模兼容测试矩阵
- GitHub Actions 或自动发布流水线
- 真实远端写入型 push execute 自动化测试

## 风险点

### 1. 文档与清单不能替代宿主验证

即使包仓库本地测试全部通过，仍然只能证明包内测试集通过。
宿主安装、命令接管与真实 API 行为仍需要宿主级验证补充把关。

### 2. 发布文档会放大当前兼容性承诺

一旦文档写得过满，后续维护成本会提高。
因此文档应坚持“已验证内容明确写、未验证内容明确保留”。

## 推荐执行顺序

1. 更新进度文档
2. 记录本轮计划与日志
3. 新增发布前检查文档
4. 运行测试验证

## 预期结果

- `todo/` 中的计划与进度重新对齐现状
- 后续 tag / 发布 / 宿主继续验证时有明确清单可跟
- 继续保持本地验证优先，不额外引入当前不需要的远端自动化

## 结论

- 状态：执行中
- 当前推荐：先做状态同步 + 发布前检查清单，并保持本地验证优先
